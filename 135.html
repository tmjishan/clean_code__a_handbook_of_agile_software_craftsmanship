104 Chapter 7: Error Handling Use Exceptions Rather Than Return Codes Back in the distant past there were many languages that didn’t have exceptions. In those languages the techniques for handling and reporting errors were limited. You either set an error ﬂag or returned an error code that the caller could check. The code in Listing 7-1 illustrates these approaches. The problem with these approaches is that they clutter the caller. The caller must check for errors immediately after the call. Unfortunately, it’s easy to forget. For this rea- son it is better to throw an exception when you encounter an error. The calling code is cleaner. Its logic is not obscured by error handling. Listing 7-2 shows the code after we’ve chosen to throw exceptions in methods that can detect errors.
